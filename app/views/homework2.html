  2. Create a Hello World in NodeJs and create a tutorial about it.

    To create a project in nodeJs first you have to make sure to have it installed. You can do this by typing node -v on your terminal, if you don't have it installed go to the official NodeJs page to install it.

    We are now going to create a simple web application platform where we simple show a "Hello World" message when you make a Http request to it.

    The code looks like this: 

    	var http = require("http");	

	var server = http.createServer(function (request, response) {
	response.writeHead(200, {'Content-Type': 'text/plain'});
	response.end("Hello World\n");
});

    The first line loads the module we need to use to create a Http server, this is done with the require function and using the module name as argument.

    The server is created with a function from the Http module using another funcion as argument, which describes how this Http server will behave. In this case we just tell it to respond the string Hello World", with a Http code of 200 (which means the request was successful, and telling whoever is making the request that the response is in plain text.

    We also need to add the next line, to indicate on which port our server is listening for requests. 

    	server.listen(8000);

    With this our server is complete, to run it locally we just need to run the command node {filename} and make a request to it, for example by entering localhost:8000 on any browser.

  5. Read and explain the post about Microservices Trade-Offs by Martin Fowler (http://martinfowler.com/articles/microservice-trade-offs.html).

    This post talks about the benefits and disadvantages of using a microservice architecture in any project and why you should or shouldn’t use it on your project. Microservices in general allow teams to set hard boundaries on every module of the project since each team only works on their own part and just has to make sure that it works by itself and responds in the way it is expected, while on a more monolithic architecture these boundaries often get blurred. On larger teams this becomes a lot more important since it helps to ease the coordination between them, with each one just having to worry about their own part.

    Another important benefit from microservices is that each module in itself becomes simpler to develop since the development team has a specific goal to achieve and they can use the technology that best works for that, mixing languages, frameworks and databases on a big team. Independent deployment is also simpler since you only test and deploy that service by itself, instead of updating a whole project.

    The ease of development and individual deployment is setback by the complexity for the operations team since they have to manage lots of services, each one being deployed or updated continuously and interacting with one another in many different ways. This asynchronous updating also brings more trouble, because it means that while one is updating, it may cause hard-to-predict errors and even harder to debug them since they are only temporal.
